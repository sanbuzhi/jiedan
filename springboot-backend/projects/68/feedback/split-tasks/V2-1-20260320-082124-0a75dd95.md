## 1. 项目技术规格
- **技术栈**：
  - 前端：Vue.js 3.4.21 + Element Plus 2.6.0 + Vue Router 4.3.0 + Pinia 2.1.7 + Axios 1.6.8 + Lodop 6.5.2.6
  - 后端：Spring Boot 2.7.18 + MyBatis Plus 3.5.5 + JJWT 0.11.5 + Jakarta Validation 3.0.2 + 支付宝Java SDK v4.38.172.ALL + 微信支付Java SDK V3 2.7.0
  - 数据库：MySQL 8.0.36
  - 部署：Docker 24.0.7 + Docker Compose 2.24.6
- **项目结构规范**：
  - 后端（单模块Spring Boot项目）：
    ```
    kids-wardrobe-pos/
    ├── src/main/java/com/kidswardrobe/pos/
    │   ├── controller/       # 控制器层
    │   ├── service/          # 服务层接口
    │   │   └── impl/         # 服务层实现
    │   ├── mapper/           # MyBatis Plus Mapper接口
    │   ├── entity/           # 数据库实体类
    │   ├── dto/              # 数据传输对象（请求参数）
    │   ├── vo/               # 视图对象（响应参数）
    │   ├── config/           # 配置类（JWT、MyBatis Plus、跨域、支付等）
    │   ├── utils/            # 工具类（加密、JWT、日期、打印、Excel等）
    │   └── exception/        # 自定义异常类 + 全局异常处理器
    ├── src/main/resources/
    │   ├── mapper/           # MyBatis Plus XML映射文件
    │   ├── db/
    │   │   ├── init.sql      # 数据库初始化脚本
    │   │   └── backup.sh     # 数据库自动备份脚本
    │   ├── application.yml   # 主配置文件
    │   ├── application-dev.yml # 开发环境配置
    │   └── application-prod.yml # 生产环境配置
    └── pom.xml
    ```
  - 前端（Vue3单页应用）：
    ```
    kids-wardrobe-pos-web/
    ├── src/
    │   ├── api/              # 后端接口封装
    │   ├── assets/           # 静态资源（Logo、样式图片等）
    │   ├── components/       # 公共组件（表头、分页、弹窗、属性选择器等）
    │   ├── views/            # 页面组件（按业务模块分组）
    │   │   ├── base/         # 基础设置模块
    │   │   ├── inventory/    # 进销存管理模块
    │   │   ├── pos/          # POS收银管理模块
    │   │   └── report/       # 流水报表管理模块
    │   ├── router/           # 路由配置
    │   ├── store/            # Pinia状态管理（用户信息、购物车、店铺配置等）
    │   ├── utils/            # 工具类（Axios拦截、日期、打印、Excel等）
    │   └── styles/           # 全局样式、POS大字体样式
    ├── public/
    │   ├── lodop/            # Lodop打印插件安装包
    │   └── favicon.ico       # 网站图标
    └── package.json
    ```
- **代码生成规则**：
  - 依赖版本：所有依赖通过pom.xml/package.json统一管理，禁止手动指定不一致的版本
  - 校验规则：所有请求参数使用Jakarta Validation/Element Plus表单校验
  - JWT配置：Token过期时间30分钟，使用16位以上随机盐值加密，支持自动刷新
  - 密码加密：使用SHA-256+16位随机盐值加密存储
- **核心业务领域锚点**：【童趣衣橱单店Web端零售进销存POS管理系统】

## 5. 业务逻辑规则
### 5.1 核心业务流程
#### 5.1.1 POS收银全流程
1. **商品录入**：支持扫码枪扫SKU、手动输入SKU/SPU多属性选品，实时校验库存（库存≤0时禁止加入购物车）
2. **购物车调整**：支持修改数量、删除商品、添加备注，实时计算总吊牌价
3. **会员识别与权益应用**：
   - 手机号/会员码识别会员，展示当前等级、累计消费、可用积分、生日信息
   - 自动判断是否符合生日折扣（生日前后N天，N可配置），弹窗确认后应用
   - 未应用生日折扣时自动应用会员等级折扣，自动判断是否符合促销活动（满减/满件折，优先选最大优惠）
   - 支持手动输入积分抵现，抵现规则为100积分抵1元（可配置），抵现金额≤可用积分对应金额≤应付金额
   - 权益叠加规则：生日折扣与会员折扣不可叠加，积分抵现可与任何折扣叠加
4. **支付结算**：
   - 支持单种/最多两种组合支付，默认按基础设置的优先级排序
   - 现金支付：输入实付金额，自动计算找零，实付不足时禁止结算
   - 第三方支付：支持扫码枪扫顾客付款码（实时调用官方SDK扣款）或生成动态收款码（实时监听官方SDK回调通知）
   - 会员卡余额支付：需先识别会员，输入支付金额，自动校验余额是否充足
   - 组合支付：需分别输入每种支付方式的金额，总金额必须等于应付金额
5. **支付成功后续**：
   - 实时扣减SKU库存（区域库存优先扣减POS默认区域，可配置），生成销售出库流水
   - 若识别会员：增加累计消费金额、按积分倍数计算并增加积分、若使用会员卡余额支付则扣减余额
   - 自动生成交易流水号、支付记录
   - 调用默认小票模板、Lodop插件打印1-3份小票
   - 记录操作日志

#### 5.1.2 采购入库全流程
1. **采购单创建**：仓库管理员/店长可创建/保存草稿，录入供应商信息、采购日期、预计到货日期、商品明细（支持扫码/手动输入SKU、输入采购数量/单价/折扣）
2. **采购单提交与审核**：提交后需超级管理员/店长审核，审核通过后方可入库；审核不通过需退回并填写原因
3. **采购入库操作**：仓库管理员核对实际到货数量（支持修改，需填写原因）、确认入库时间
4. **入库成功后续**：
   - 实时增加SKU总库存、采购单关联区域库存（默认仓库区域，可配置），生成采购入库流水
   - 更新SKU的最近采购价
   - 记录操作日志

#### 5.1.3 库存盘点全流程
1. **盘点任务创建**：超级管理员/店长可创建全店/区域/品类/SKU指定盘点任务，分配盘点人、录入盘点日期
2. **盘点操作**：盘点人可通过扫码枪/手动输入SKU盘点，输入实际库存数量，系统自动对比账面库存生成盘盈/盘亏/账实相符明细，支持保存草稿
3. **盘点结果提交与审核**：提交后需超级管理员/店长审核，审核通过后方可调整库存；审核不通过需退回并填写原因
4. **库存调整成功后续**：
   - 实时调整SKU总库存，生成盘盈/盘亏入库/出库流水
   - 计算盘盈/盘亏金额（盘盈金额=吊牌价×盘盈数量，盘亏金额=最近采购价×盘亏数量）
   - 记录操作日志

#### 5.1.4 库存预警触发流程
1. **库存变动后自动检测**：每次采购入库/销售出库/盘盈盘亏/调拨后，系统自动检测所有SKU的库存
2. **阈值匹配**：优先匹配SKU级阈值，无SKU级阈值则匹配品类级阈值，无阈值则跳过
3. **预警消息生成**：
   - 当SKU库存≤最低阈值时，生成“库存不足预警”消息，推送给后台首页、仓库管理员/店长的消息中心
   - 当SKU库存≥最高阈值（若设置）时，生成“库存积压预警”消息，推送给后台首页、仓库管理员/店长的消息中心
4. **预警消息处理**：处理人可点击预警消息处理，填写处理备注，处理后消息状态变为“已处理”

### 5.2 数据校验规则
1. **SKU字典校验**：
   - 品类编码唯一、品类名称唯一
   - 属性模板必须绑定品类，属性必须绑定模板
   - SPU编码唯一、SKU编码唯一（自动生成规则为：2位品类编码+4位SPU编码+属性编码组合，属性不足时补0）
   - SKU必须有唯一的属性值组合
2. **会员体系校验**：
   - 会员手机号唯一（加密存储，校验时先解密前缀+后缀）
   - 会员等级的累计消费金额必须递增（后一等级的累计消费金额>前一等级）
   - 积分调整必须填写原因，仅超级管理员/店长可操作
3. **采购/退货/调拨/盘点单据校验**：
   - 单据编码唯一（自动生成规则为：前缀（PC/PY/TB/PD）+8位日期+4位流水号）
   - 采购/退货/调拨/盘点的商品明细不能为空
   - 采购单价/折扣后单价必须≥0，采购/退货/调拨/盘点数量必须≥1
   - 销售退货必须关联原交易流水号，退货数量≤原销售数量-已退货数量
   - 超过退货时间限制的退货单（可配置7天/30天），需店长以上级别审核才可提交
4. **POS收银校验**：
   - 购物车商品不能为空，商品数量必须≥1，库存必须≥商品数量
   - 积分抵现金额必须≤可用积分对应金额≤应付金额
   - 单种/组合支付的总实付金额必须≥应付金额（现金支付可≥，组合支付必须=）
   - 会员卡余额支付的金额必须≤会员卡余额
5. **通用数据校验**：
   - 所有日期格式必须为YYYY-MM-DD或YYYY-MM-DD HH:mm:ss
   - 所有金额格式必须为小数点后2位，必须≥0
   - 所有必填字段不能为空

### 5.3 状态流转规则
#### 5.3.1 采购单状态流转
DRAFT（草稿）→ SUBMITTED（已提交）→ AUDITED（已审核）→ FULL_IN（已入库）；DRAFT→可编辑删除；SUBMITTED→可撤回至DRAFT或审核；AUDITED→可执行入库；FULL_IN→不可修改删除
#### 5.3.2 销售单状态流转
UNPAID（未支付）→ PAID（已支付）→ PARTIAL_RETURN（部分退货）/FULL_RETURN（已退货）；UNPAID→可编辑删除；PAID→可退货；PARTIAL_RETURN→可继续退货至FULL_RETURN；FULL_RETURN→不可修改删除
#### 5.3.3 支付记录状态流转
PENDING（待支付）→ SUCCESS（成功）/FAILED（失败）；SUCCESS→可退款；REFUNDING（退款中）→ REFUNDED（已退款）
#### 5.3.4 预警消息状态流转
UNHANDLED（未处理）→ HANDLED（已处理）；UNHANDLED→可点击处理；HANDLED→不可修改

### 5.4 异常处理规则
1. **第三方支付异常处理**：
   - 调用第三方支付接口超时（≥5秒）时，提示“支付超时，请稍后查询交易状态”，同时将支付记录状态设为PENDING，后续可通过定时任务（每5分钟执行一次）或手动查询第三方支付接口更新支付状态
   - 第三方支付接口返回失败时，提示“支付失败，请重新支付”，同时将支付记录状态设为FAILED
2. **库存不足异常处理**：
   - POS收银加入购物车时库存不足：提示“库存不足，请调整数量或更换商品”，禁止加入购物车
   - POS收银支付成功后扣减库存时库存不足：立即回滚所有操作（销售单、支付记录、会员权益变动），提示“库存扣减失败，请稍后重新支付”
3. **数据修改/删除异常处理**：
   - 所有涉及已完成业务的单据/SPU/SKU/会员等级/支付方式，禁止修改/删除（仅可下架SPU、禁用会员/支付方式）
   - 删除角色前需先转移该角色下的所有用户，否则提示“请先转移该角色下的所有用户”
4. **全局异常处理**：
   - 所有未捕获的异常均由全局异常处理器捕获，返回统一的错误响应格式（错误码、错误信息），并记录操作日志
   - 敏感操作（如密码重置、积分调整、数据备份）失败时，必须记录详细的错误日志

## 7. 代码生成规范
### 7.1 命名规范
1. **核心业务领域标识**：所有类名、方法名、变量名、表名、字段名、错误码前缀均包含「kids」（童趣衣橱的英文缩写Kids' Wardrobe）
2. **类名**：采用大驼峰命名法（PascalCase），格式为Kids+业务模块+功能类，例如：KidsUserController、KidsSkuService、KidsSalesOrder、KidsPasswordUtil
3. **方法名**：采用小驼峰命名法（camelCase），格式为动词+业务对象+动作（可选），例如：saveKidsRole、getKidsSkuByCode、auditKidsPurchaseOrder、calculateKidsSalesDiscount
4. **变量名**：采用小驼峰命名法（camelCase），格式为业务对象+属性（可选），例如：kidsUserId、kidsSkuCode、kidsTotalSalesAmount
5. **表名**：采用小写下划线命名法（snake_case），格式为kids_+业务模块+功能表，例如：kids_user、kids_sku、kids_sales_order、kids_operation_log
6. **字段名**：采用小写下划线命名法（snake_case），格式为业务对象+属性（可选），例如：kids_user_id、kids_sku_code、total_sales_amount
7. **错误码**：采用大写下划线命名法（UPPER_SNAKE_CASE），格式为KIDS_+业务模块_+错误序号（3位数字），例如：KIDS_USER_001（用户名已存在）、KIDS_SKU_001（SKU编码已存在）、KIDS_PAY_001（支付失败）

### 7.2 代码结构
1. **后端分层**：严格遵循Controller→Service→Mapper三层架构
   - Controller层：仅负责接收请求参数、校验参数、调用Service层、返回统一响应格式
   - Service层：负责所有核心业务逻辑、事务管理（@Transactional注解）、调用Mapper层
   - Mapper层：仅负责数据库操作（继承MyBatis Plus的BaseMapper，复杂SQL写在XML映射文件中）
2. **前端分层**：严格遵循views→components→api→store→utils五层架构
   - views层：仅负责页面布局、组件调用、数据展示
   - components层：负责公共组件/业务组件的封装
   - api层：负责所有后端接口的封装（统一使用Axios，添加请求/响应拦截器）
   - store层：负责全局状态管理（Pinia，如用户信息、购物车、店铺配置）
   - utils层：负责工具类的封装（Axios拦截、日期、打印、Excel、权限判断）

### 7.3 注释要求
1. **类注释**：所有类（Controller/Service/Mapper/Entity/DTO/VO/Util/Config）必须添加类注释，说明类的功能、作者、创建时间
2. **方法注释**：所有公共方法（public）必须添加方法注释，说明方法的功能、请求参数、响应格式、业务逻辑、作者、创建时间
3. **字段注释**：所有数据库实体类的字段、所有DTO/VO的字段必须添加字段注释，说明字段的含义、约束、默认值
4. **复杂业务逻辑注释**：所有复杂的业务逻辑（如POS收银权益计算、库存盘点盘盈盘亏计算）必须添加行内注释，说明每一步的逻辑

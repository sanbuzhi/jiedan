# TASKS.md - 童趣衣橱单店进销存POS管理系统

## 1. 项目技术规格
- **技术栈**：
  - 前端：Vue 3.4.21 + Element Plus 2.5.6 + Axios 1.6.8 + Pinia 2.1.7 + Vue Router 4.3.0
  - 后端：Spring Boot 2.7.18 + MyBatis Plus 3.5.5 + MySQL 8.0.36 + JWT 0.11.5 + Flyway 9.22.3 + Hutool 5.8.26
  - 第三方：支付宝SDK Java 4.39.85.ALL + 微信支付SDK Java 0.2.12 + Lodop 6.7.0.2 + Docker 24.0.7
- **项目结构规范**：
  - 前端目录：`/frontend` → `src/components`（公共+POS大组件）、`src/views`（基础设置/进销存/POS/报表/首页/登录）、`src/api`（按模块分接口）、`src/router`、`src/store`（用户/购物车/打印）、`src/utils`、`public`（静态资源）
  - 后端目录：`/backend` → `src/main/java/com/kidscloset`（controller/service/mapper/entity/dto/vo/config/exception/task/utils）、`src/main/resources`（application.yml、db/migration/Flyway脚本）
  - 数据库脚本：存放在`/backend/src/main/resources/db/migration`，按版本号命名（V1.0.0_init_schema.sql、V1.0.1_init_data.sql）
- **代码生成规则**：
  - 依赖版本统一使用技术栈指定版本
  - 数据校验：使用Hibernate Validator（手机号正则`^1[3-9]\d{9}$`、金额≥0.01、数量≥1整数等）
  - JWT配置：密钥使用SHA-256随机生成，过期时间30分钟，携带用户ID、角色编码
  - 第三方支付超时：5秒，超时后交易状态设为「待确认」，支持后续主动查询
- **核心业务领域锚点**：【童趣衣橱单店进销存POS管理系统】

## 6. 业务逻辑规则
### 核心业务流程
1. **童趣衣橱SKU进销存全链路流程**：
   - 创建品类→属性模板→SPU→自动生成SKU→设置库存预警阈值→采购单创建/提交/审核→采购入库（增加库存、生成流水）→POS收银（扣减库存、生成流水）→销售退货申请/审核→退款入库（增加库存、生成流水）→库存盘点（对比账面/实际、调整库存、生成流水）
2. **童趣衣橱POS收银流程**：
   - 登录POS收银台→商品录入（扫码/手动/多属性）→库存校验→购物车管理→会员识别→权益自动应用（会员/生日折扣、积分抵现、促销）→支付方式选择→组合支付（可选）→结算→支付成功→即时打印小票
3. **童趣衣橱交接班流程**：
   - 收银员点击交接班→系统统计当前班次数据→手动输入当前备用金→计算备用金差额→生成交接班报表→打印报表→下一班/店长确认→生效

### 数据校验规则
1. **SKU相关校验**：
   - SKU唯一标识必须唯一，规则为：品类编码（2位）+SPU编码（6位）+属性编码（每个属性最多2位，拼接）
   - 加入购物车时，SKU可用库存必须≥购买数量
2. **会员相关校验**：
   - 手机号必须唯一，格式为`^1[3-9]\d{9}$`，AES加密存储
   - 积分抵现金额不得超过可用积分对应的金额，也不得超过应收金额
3. **支付相关校验**：
   - 现金支付实付金额必须≥应收金额
   - 组合支付总金额必须等于应收金额
4. **进销存相关校验**：
   - 采购单审核通过后才能入库
   - 销售退货必须关联原交易流水号，且退货数量≤原销售数量-已退货数量
   - 超过基础设置中退货时间的退货单，需店长以上级别审核才可提交

### 状态流转规则
1. **采购单状态**：
   - DRAFT（草稿）→SUBMITTED（已提交）→AUDITED_PASS（审核通过）/AUDITED_REJECT（审核驳回）→DRAFT（驳回后可重新编辑）
   - AUDITED_PASS→PENDING（待入库）→PARTIAL_INBOUND（部分入库）→FULL_INBOUND（全部入库）
2. **销售单状态**：
   - NORMAL（正常销售）→PARTIAL_RETURNED（部分退货）→FULL_RETURNED（全部退货）
3. **盘点任务状态**：
   - DRAFT（草稿）→SUBMITTED（已提交）→AUDITED_PASS（审核通过）/AUDITED_REJECT（审核驳回）→DRAFT（驳回后可重新编辑）
   - AUDITED_PASS→PENDING（待调整）→ADJUSTED（已调整）

### 异常处理规则
1. **第三方支付超时**：
   - 交易状态设为「待确认」，推送消息给收银员/店长，支持后续主动查询第三方支付接口
2. **库存不足**：
   - 加入购物车时弹出提示，阻止继续结算
3. **数据重复**：
   - 捕获数据库唯一约束异常，返回友好提示（如「该SKU唯一标识已存在」「该手机号已注册会员」）
4. **权限不足**：
   - 未授权的功能模块/操作按钮隐藏，直接访问接口返回403错误

## 8. 代码生成规范
### 命名规范
- **类名**：大驼峰命名，必须包含核心业务领域标识前缀「Kidscloset」，如KidsclosetRoleController、KidsclosetSkuService
- **方法名**：小驼峰命名，必须清晰表达业务意图，如saveKidsclosetRole、getKidsclosetSkuByCode
- **变量名**：小驼峰命名，禁止使用拼音，如skuCode、totalSalesAmount
- **表名**：小写字母+下划线命名，必须包含核心业务领域标识前缀「kidscloset_」，如kidscloset_role、kidscloset_sku
- **字段名**：小写字母+下划线命名，如sku_code、total_tag_price
- **错误码**：必须包含核心业务领域标识前缀「KC_」，如KC_001（用户未登录）、KC_002（权限不足）、KC_101（库存不足）

### 代码结构
- **前端**：严格按模块分层（components/views/router/store/api/utils），POS专用组件单独放在`src/components/pos`目录
- **后端**：严格按MVC+Service分层（controller/service/mapper/entity/dto/vo/config/exception/task/utils），每个业务模块单独分包

### 注释要求
- **类注释**：必须包含类的功能描述、作者、创建时间
- **方法注释**：必须包含方法的功能描述、请求参数、响应格式、业务逻辑、异常处理
- **复杂业务逻辑**：必须添加行内注释，解释关键步骤
- **数据库表/字段**：必须在Flyway脚本中添加清晰的注释

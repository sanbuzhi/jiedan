## 1. 项目技术规格
### 技术栈
- **前端技术栈**：
  店长端：React 18.x + Ant Design 5.x + Redux Toolkit 2.x + React Router 6.x + Vite 4.x + ESLint + Prettier
  客户端：React 18.x + Vant 4.x + Zustand 4.x + React Router 6.x + Vite 4.x + ESLint + Prettier
- **后端技术栈**：
  Node.js 18.x + NestJS 9.x + TypeORM 0.3.x + MySQL 8.x + Redis 7.x + class-validator + class-transformer + JWT
- **第三方集成**：
  微信JSAPI/Native支付、支付宝手机/电脑网站支付、阿里云短信服务、阿里云OSS
- **部署技术**：
  Docker + Docker Compose + Nginx + Let's Encrypt SSL证书 + Prometheus + Grafana

### 代码生成规则
- **依赖版本**：严格按照技术栈指定的大版本锁定（使用npm shrinkwrap或yarn.lock）
- **校验规则**：
  - 后端：使用class-validator校验DTO参数，自定义儿童服装相关校验（如尺码格式、效期合法性、价格非负等）
  - 前端：使用AntD/Vant内置表单校验+自定义正则表达式
- **JWT配置**：
  - 访问令牌（access_token）：店长端有效期7天，客户端有效期3天，算法HS256
  - 刷新令牌（refresh_token）：店长端有效期30天，客户端有效期15天，仅用于刷新访问令牌
  - 密钥：使用随机生成的32位以上字符串存储在环境变量中
- **文件上传规则**：
  - 商品图片：主图/轮播图≤500KB（压缩至750×750/750×1000px），详情图≤200KB，格式仅支持JPG、PNG、WEBP
  - 商品视频：≤100MB，转码为MP4 720P 2Mbps，时长≤30秒
  - 临时文件（Excel导入导出等）：存储在OSS临时目录，有效期24小时

## 5. 业务逻辑规则
### 核心业务流程
#### 1. 儿童服装采购入库流程
1. 店长在入库管理模块创建「采购入库单」，选择商品SKU、填写数量和进价
2. 提交入库单前可修改，提交后系统自动校验库存状态（非盘点中）
3. 校验通过后，更新对应商品SKU的库存数量，生成入库操作日志
4. 自动检查库存是否超过最高预警值，若超过则生成滞销预警消息

#### 2. 儿童服装线上订单流程
1. 客户在客户端浏览商品，加购或立即购买
2. 结算时选择配送方式、自提时间/收货地址、优惠券、支付方式
3. 提交订单后系统自动计算费用、锁定商品SKU库存、生成订单号和自提验证码（若自提）
4. 客户需在30分钟内完成支付，超过30分钟未支付订单自动取消，释放库存
5. 支付成功后，订单状态变为「待发货/待自提」，自动向客户发送支付成功短信
6. 店长处理订单：本地配送需24小时内发货，自提需客户到店核销
7. 订单完成后，更新会员累计消费金额和次数，自动检查是否升级会员等级

#### 3. 会员优惠券发放核销流程
1. 店长创建优惠券，设置发放规则、使用规则、有效期
2. 手动或自动发放优惠券给符合条件的会员，发送发放成功短信
3. 客户在结算时选择符合条件的优惠券，系统自动校验使用范围、门槛、有效期、等级限制、是否已使用
4. 校验通过后核销优惠券，更新优惠券使用状态和剩余发放数量，生成核销操作日志

### 数据校验规则
#### 1. 儿童服装商品校验
- 尺码格式：必须为66/73/80/90/100/110/120/130/140/150码或自定义的数字+「码」格式
- 价格校验：进价、售价必须≥0，售价必须≥进价
- 库存校验：入库数量、出库数量必须≥1，库存数量不能为负数

#### 2. 会员校验
- 手机号格式：必须为中国大陆11位手机号
- 会员卡号格式：必须为TQ202开头的12位字符串
- 生日格式：必须为YYYY-MM-DD格式

#### 3. 订单校验
- 自提时间：必须在门店自提时间范围内，且在原自提时间前后2小时内（若修改）
- 本地配送地址：必须在配置的配送范围内
- 支付金额：必须与系统计算的实付金额一致

### 状态流转规则
#### 1. 儿童服装订单状态流转
```
pending_pay → [支付成功] → pending_delivery/pending_self_pickup
pending_pay → [超时未支付/手动取消] → cancelled
pending_delivery → [店长发货] → delivered
delivered → [客户确认收货/7天未异议] → completed
pending_self_pickup → [客户到店核销] → completed
pending_self_pickup → [7天未自提] → cancelled
completed → [客户申请售后/7天内] → refunding
refunding → [审核通过仅退款] → refunded
refunding → [审核通过退货退款] → pending_return
pending_return → [店长确认收货] → refunding
refunding → [退款成功] → refunded
refunding → [审核拒绝/拒绝收货] → completed
```

#### 2. 会员优惠券状态流转
```
not_started → [到达发放时间] → ongoing
ongoing → [停止发放] → stopped
ongoing → [到达有效期结束时间] → expired
stopped → [到达有效期结束时间] → expired
```

### 异常处理规则
#### 1. 支付异常处理
- 支付回调超时或失败：系统每5分钟自动查询支付状态，重试3次，3次后仍失败则标记为「支付异常」，通知店长处理
- 支付金额与订单实付金额不一致：系统拒绝处理回调，标记为「支付异常」，通知店长处理

#### 2. 库存异常处理
- 入库/出库时库存状态为「盘点中」：系统拒绝操作，提示「正在盘点中，请稍后再试」
- 提交订单时库存不足：系统提示「商品库存不足，请联系店长或选择其他SKU」

#### 3. 短信异常处理
- 短信发送失败：系统记录失败日志，支持手动重发
- 短信发送频率超过限制：系统提示「发送太频繁，请稍后再试」

## 7. 代码生成规范
### 命名规范
#### 1. 类名/接口名/枚举名
- 采用大驼峰命名法（PascalCase）
- 必须包含核心业务领域关键词「Tongqu」或「ChildClothing」（尽量使用「Tongqu」）
- 示例：TongquUserController、TongquProductSkuService、TongquOrderStatusEnum

#### 2. 方法名/变量名
- 采用小驼峰命名法（camelCase）
- 尽量使用核心业务领域相关词汇
- 示例：createChildClothingSku、getTongquMemberList、stockInQuantity

#### 3. 表名/字段名
- 采用小写下划线命名法（snake_case）
- 表名前缀必须为「tongqu_」
- 尽量使用核心业务领域相关词汇
- 示例：tongqu_product_sku、tongqu_member_coupon、child_clothing_size

#### 4. 接口URL
- 采用小写下划线命名法（snake_case）
- 前缀必须为「/api/tongqu/」
- 店长端接口前缀为「/api/tongqu/admin/」，客户端接口前缀为「/api/tongqu/client/」
- 尽量使用RESTful风格
- 示例：/api/tongqu/admin/inventory/sku/add、/api/tongqu/client/product/detail/:id

### 错误码定义
- 错误码格式为「TQ_XXX_XXX」，其中：
  - 前缀「TQ」为核心业务领域标识
  - 中间三位为模块标识（001公共模块、002系统基础模块、003进销存管理模块、004会员管理模块、005订单管理模块、006数据报表模块）
  - 最后三位为具体错误标识
- 示例：
  - TQ_001_001：参数校验失败
  - TQ_001_002：身份认证失败
  - TQ_003_001：商品SKU不存在
  - TQ_003_002：库存不足
  - TQ_005_001：订单不存在
  - TQ_005_002：订单状态不正确

## 1. 项目技术规格
### 技术栈
- **前端技术栈**：React 18.17LTS + Ant Design 5.12.x（店长端）、Vant 4.8.x（客户端）、Redux Toolkit 2.0.x（店长端状态管理）、Zustand 4.4.x（客户端状态管理）、React Router 6.20.x、Vite 4.5.x、ESLint + Prettier
- **后端技术栈**：Node.js 18.17LTS、NestJS 9.4.x、TypeORM 0.3.17、MySQL 8.0.35、Redis 7.2.4、JWT（jsonwebtoken 9.0.2）、Swagger/OpenAPI 3.0（@nestjs/swagger 7.1.17）、ClassValidator + ClassTransformer
- **第三方集成**：微信JSAPI/Native支付、支付宝手机网站/电脑网站支付、阿里云短信服务、阿里云OSS、微信内置支付SDK适配
- **部署工具**：Docker 24.x、Docker Compose 2.23.x、Nginx 1.25.x、Let's Encrypt SSL证书

### 代码生成规则
- **依赖版本**：严格按照技术栈中指定的LTS或稳定版本
- **校验规则**：
  - 后端：使用ClassValidator进行请求参数校验，ClassTransformer进行类型转换
  - 前端：使用Zustand/Redux Toolkit中的中间件或自定义逻辑进行简单本地校验，复杂校验调用后端接口
- **JWT配置**：
  - 访问令牌（access_token）：店长端24小时，客户端2小时
  - 刷新令牌（refresh_token）：店长端7天，客户端30天
  - 密钥：通过环境变量配置，生产环境密钥长度≥32位
  - 加密算法：HS256
- **文件压缩转码**：前端上传前压缩图片，后端使用sharp压缩、ffmpeg转码视频
- **数据库连接池**：TypeORM连接池大小配置为20（生产环境）、10（开发/测试环境）
- **Redis配置**：缓存过期时间：手机验证码5分钟，购物车7天，商品库存1分钟，热门商品列表30分钟

## 3. 后端接口开发清单
### 模块1：认证模块
| URL | 请求方式 | 请求参数 | 响应格式 | 业务逻辑 | 关联数据库表 |
|-----|----------|----------|----------|----------|--------------|
| /api/auth/send-sms | POST | phone（必填）、type（login/register/reset-password/modify-phone，必填） | {code:200,msg:"发送成功",data:null} | 1. 校验手机号格式<br>2. 校验发送频率（每分钟1次）<br>3. 生成6位随机验证码<br>4. 存入Redis（过期5分钟）<br>5. 调用阿里云短信服务发送 | sms_log |
| /api/auth/register | POST | phone（必填）、code（必填）、password（可选，默认123456）、name（可选）、gender（可选）、birthday（可选） | {code:200,msg:"注册成功",data:{access_token,refresh_token,memberInfo}} | 1. 校验手机号格式、验证码<br>2. 校验手机号是否已存在<br>3. 生成会员卡号（TQ202XXXXXX）<br>4. 密码SHA-256加密<br>5. 初始化会员等级为普通会员<br>6. 存入数据库<br>7. 发送注册成功短信<br>8. 生成并返回JWT | member_info、sms_log、member_level_config |
| /api/auth/login | POST | phone/username（必填）、password/code（必填）、type（manager/customer，必填） | {code:200,msg:"登录成功",data:{access_token,refresh_token,userInfo}} | 1. 校验身份类型<br>2. 校验账号/手机号格式、密码/验证码<br>3. 校验账号是否锁定（连续3次错误锁定15分钟）<br>4. 生成JWT<br>5. 更新最后登录时间 | manager_info、member_info、sms_log、login_log |
| /api/auth/logout | POST | refresh_token（Header） | {code:200,msg:"退出成功",data:null} | 1. 校验refresh_token<br>2. 将refresh_token加入Redis黑名单（过期时间=原refresh_token剩余时间） | - |

## 5. 业务逻辑规则
### 核心业务流程
#### 5.1 采购入库流程
1. 店长在店长端创建采购入库单，选择供应商、入库日期、入库商品SKU、数量、进价
2. 提交入库单前可修改，提交后系统自动生成入库单号
3. 系统自动更新对应商品SKU的库存数量
4. 系统自动检查更新后库存的状态，生成或更新库存预警
5. 系统自动清除相关商品的Redis缓存
6. 系统自动记录操作日志
#### 5.2 客户线上购买流程
1. 客户在客户端浏览/搜索商品，选择SKU加入购物车或立即购买
2. 进入结算页面，选择配送方式（自提/本地配送）、自提时间/收货地址、优惠券、支付方式
3. 提交订单后系统自动生成订单号，锁定对应商品SKU的库存数量（30分钟）
4. 跳转到第三方支付页面，客户完成支付
5. 第三方支付回调/系统主动查询支付状态，若支付成功：
   a. 更新订单状态为「待发货/待自提」
   b. 自动扣减锁定的商品SKU库存
   c. 自动检查更新后库存的状态，生成或更新库存预警
   d. 自动清除相关商品的Redis缓存
   e. 自动使用符合条件的优惠券，更新优惠券状态
   f. 自动更新会员的累计消费金额、累计消费次数、最后消费时间
   g. 后台异步任务自动重新计算会员等级
   h. 自动发送支付成功短信
   i. 自动记录操作日志
6. 若30分钟内未支付，系统自动取消订单，释放锁定的库存
#### 5.3 自提订单核销流程
1. 系统在自提时间前1小时自动向客户发送自提提醒短信（包含自提验证码）
2. 客户到店自提，店长在店长端输入自提验证码或扫描会员二维码
3. 系统验证自提验证码/会员二维码的有效性
4. 验证通过后：
   a. 更新订单状态为「已完成」
   b. 自动生成销售出库单
   c. 自动发送自提核销成功短信
   d. 自动记录操作日志
#### 5.4 退货退款流程
1. 客户在已完成订单的详情页提交退货退款申请，填写原因、金额、上传图片（可选）
2. 系统向店长端推送退款售后提醒消息
3. 店长在24小时内审核申请：
   a. 若审核通过：
      i. 更新退款售后状态为「审核通过」，若为「仅退款」则直接更新为「退款中」并调用第三方退款接口
      ii. 若为「退货退款」，更新状态为「待退货」，发送审核通过短信（包含退货地址、截止时间）
   b. 若审核拒绝：
      i. 更新退款售后状态为「审核拒绝」
      ii. 恢复订单状态为「已完成」
      iii. 发送审核拒绝短信
4. 若为「退货退款」且审核通过，客户在7天内退货到店：
   a. 店长检查商品是否完好无损、符合退货条件
   b. 若检查通过：
      i. 更新退款售后状态为「已退货」，然后更新为「退款中」并调用第三方退款接口
      ii. 若选择重新上架，自动生成退货入库单
   c. 若检查不通过：
      i. 更新退款售后状态为「审核拒绝」
      ii. 恢复订单状态为「已完成」
      iii. 发送拒绝收货短信
5. 第三方退款回调/系统主动查询退款状态，若退款成功：
   a. 更新退款售后状态为「退款完成」
   b. 更新订单状态为「退款完成」
   c. 若使用了优惠券，自动恢复优惠券状态为「可用」（需检查有效期）
   d. 自动更新会员的累计消费金额、累计消费次数
   e. 后台异步任务自动重新计算会员等级
   f. 自动发送退款成功短信
   g. 自动记录操作日志
#### 5.5 库存周转&滞销预警流程
1. 后台每日凌晨3点自动计算所有商品SKU的库存周转天数
2. 库存周转天数计算公式（简化版）：`库存周转天数 = 30 / (过去30天该SKU的销售额 / 当前库存金额)`
3. 滞销库存定义：连续30天未销售的商品SKU，或库存周转天数>全局最高预警阈值配置的天数（默认45天）
4. 若商品SKU满足滞销条件且当前无未处理的滞销预警，系统自动生成滞销预警
5. 店长可在库存预警模块查看滞销预警，点击查看销售数据或手动标记为已关注

### 数据校验规则
#### 5.6 通用校验规则
1. 所有金额字段必须≥0，保留2位小数
2. 所有数量字段必须≥0，整数类型
3. 所有手机号必须符合中国大陆手机号格式（11位数字，1开头）
4. 所有日期格式必须符合YYYY-MM-DD，时间格式必须符合YYYY-MM-DD HH:mm:ss
5. 所有JSON格式的字段必须符合JSON规范
6. 所有图片/视频文件上传前必须校验类型、大小：
   - 图片类型：JPG/JPEG/PNG/GIF/WebP，主图轮播图≤500KB，详情页图片≤200KB
   - 视频类型：MP4，分辨率≤720P，码率≤2Mbps，时长≤30秒，大小≤100MB
#### 5.7 商品校验规则
1. 商品分类最多3级，删除分类时必须先转移或删除该分类下的所有子分类和商品
2. 每个商品详情必须关联至少1个SKU，SKU编码必须唯一
3. 商品主图最多5张，轮播图最多10张，标签最多5个
4. 商品标题最多30字，副标题最多50字
5. 会员折扣比例必须在0.1-0.99之间（或1.00表示不折扣）
#### 5.8 进销存校验规则
1. 入库/出库数量必须>0
2. 出库时必须校验商品SKU的库存数量≥出库数量
3. 盘点时必须先锁定商品SKU的库存状态为「盘点中」，禁止入库、出库操作
4. 库存预警阈值的最小值必须<最大值
#### 5.9 会员校验规则
1. 会员手机号必须唯一，密码必须符合强度要求（至少8位，包含字母、数字、特殊字符中的两种）
2. 会员等级最多5级，优先级必须唯一
3. 优惠券发放数量必须≥0，每人限领数量必须≥1
4. 优惠券有效期的开始日期必须<结束日期
#### 5.10 订单校验规则
1. 待支付订单的有效时间为30分钟，超过30分钟自动取消
2. 本地配送地址必须在系统配置的配送范围内
3. 自提时间必须在系统配置的自提时间范围内
4. 修改自提时间必须在原自提时间前后2小时内
5. 申请退款/售后必须在确认收货后7天内
6. 退款金额必须≤实付金额

### 状态流转规则
#### 5.11 订单状态流转规则
```
待支付 → 待发货/待自提（支付成功）
待支付 → 已取消（30分钟未支付/店长手动取消）
待发货/待自提 → 已发货/待自提确认（本地配送店长发货）
待发货/待自提 → 已取消（客户/店长手动取消）
已发货/待自提确认 → 已完成（本地配送：客户确认收货/确认送达后7天未提出异议；自提：店长核销）
已发货/待自提确认 → 退款/售后中（客户申请退款/售后）
已完成 → 退款/售后中（客户申请退款/售后，且在7天内）
退款/售后中 → 退款完成（退款成功）
退款/售后中 → 已完成（审核拒绝/退货拒绝）
退款/售后中 → 待退货（审核通过，退货退款）
待退货 → 退款中（客户退货到店，店长确认接受）
退款中 → 退款完成（退款成功）
```
#### 5.12 优惠券模板状态流转规则
```
未开始发放 → 已开始发放（达到发放条件或店长手动开始）
未开始发放 → 已删除（店长删除）
已开始发放 → 已停止发放（店长手动停止）
已停止发放 → 无法恢复
```
#### 5.13 会员优惠券状态流转规则
```
可用 → 已使用（订单支付成功）
可用 → 已过期（达到有效期结束日期，后台定时任务自动更新）
已使用 → 无法恢复
已过期 → 无法恢复
已使用 → 可用（订单退款成功，且在有效期内）
```

### 异常处理规则
#### 5.14 通用异常处理规则
1. 所有后端接口必须统一返回JSON格式的响应，包含code、msg、data三个字段
2. 所有异常必须被全局异常过滤器捕获，返回对应的错误码和错误信息
3. 所有错误日志必须被记录到日志文件和/或数据库中
4. 所有第三方集成（支付、短信、OSS）的异常必须被捕获，记录日志，并返回友好的错误信息给用户
#### 5.15 支付异常处理规则
1. 若支付回调超时或失败，系统必须在30分钟内重试3次主动查询支付状态
2. 若主动查询3次后仍无法获取支付状态，系统必须将订单状态标记为「待确认」，并向店长端推送提醒消息
3. 若退款回调超时或失败，系统必须在24小时内重试3次主动查询退款状态
#### 5.16 短信异常处理规则
1. 若短信发送失败，系统必须记录日志，并支持店长在短信日志模块手动重发
2. 若短信发送频率超过限制（每分钟1次），系统必须返回友好的错误信息给用户
#### 5.17 库存异常处理规则
1. 若扣减库存时发现库存不足，系统必须返回友好的错误信息给用户，并取消订单或出库操作
2. 若库存状态异常（如同时有两个入库单操作同一个SKU），系统必须使用乐观锁或悲观锁保证数据一致性

## 7. 代码生成规范
### 命名规范
#### 7.1 后端命名规范
- **类名**：大驼峰命名法（PascalCase），必须包含核心业务领域关键词「童趣小筑」的缩写「TQ」或直接包含关键词（如TQAuthController、TQProductService、TQMemberCouponEntity）
- **方法名**：小驼峰命名法（camelCase），必须清晰表达业务含义（如loginWithPhone、createProductSku、handleRefundAudit）
- **变量名**：小驼峰命名法（camelCase），必须清晰表达变量含义（如memberInfo、orderNo、skuCode）
- **常量名**：全大写下划线命名法（UPPER_SNAKE_CASE），必须包含核心业务领域标识「TQ」（如TQ_JWT_ACCESS_TOKEN_EXPIRE_TIME、TQ_MEMBER_NO_PREFIX）
- **表名**：全小写下划线命名法（lower_snake_case），必须包含核心业务领域关键词的前缀「tq_」（如tq_manager_info、tq_product_sku、tq_order_info）
- **字段名**：全小写下划线命名法（lower_snake_case），必须清晰表达字段含义（如member_no、order_no、sku_code）
#### 7.2 前端命名规范
- **组件名**：大驼峰命名法（PascalCase），必须包含核心业务领域关键词的前缀「TQ」或直接包含关键词（如TQManagerLayout、TQProductSkuTable、TQCustomerCart）
- **文件/文件夹名**：小驼峰命名法（camelCase）或大驼峰命名法（PascalCase，组件文件），必须清晰表达文件/文件夹含义（如src/pages/manager/product/sku/index.tsx、src/components/common/TQTable.tsx）
- **方法名**：小驼峰命名法（camelCase），必须清晰表达业务含义（如handleLogin、handleAddProductSku、handleSubmitOrder）
- **变量名**：小驼峰命名法（camelCase），必须清晰表达变量含义（如memberInfo、orderNo、skuCode）
- **常量名**：全大写下划线命名法（UPPER_SNAKE_CASE），必须包含核心业务领域标识「TQ」（如TQ_MANAGER_BASE_URL、TQ_CUSTOMER_BASE_URL、TQ_JWT_ACCESS_TOKEN_KEY）
- **状态名**：小驼峰命名法（camelCase），必须清晰表达状态含义（如Redux Toolkit的authState、productSkuList，Zustand的cartState、favoriteState）

### 错误码定义
#### 7.3 错误码前缀
所有错误码必须包含核心业务领域标识「TQ」作为前缀
#### 7.4 错误码分类
- **TQ0000**：成功
- **TQ1000-TQ1999**：系统通用错误
  - TQ1001：参数校验失败
  - TQ1002：未授权访问
  - TQ1003：权限不足
  - TQ1004：资源不存在
  - TQ1005：系统内部错误
  - TQ1006：第三方服务调用失败
  - TQ1007：文件上传失败
  - TQ1008：文件下载失败
- **TQ2000-TQ2999**：认证模块错误
  - TQ2001：账号或密码错误
  - TQ2002：手机号格式错误
  - TQ2003：手机号已注册
  - TQ2004：手机号未注册
  - TQ2005：验证码错误
  - TQ2006：验证码已过期
  - TQ2007：验证码发送频率过高
  - TQ2008：账号已锁定
  - TQ2009：旧密码错误
  - TQ2010：新密码强度不足
- **TQ3000-TQ3999**：商品模块错误
  - TQ3001：商品分类不存在
  - TQ3002：商品分类下有子分类或商品，无法删除
  - TQ3003：商品详情不存在
  - TQ3004：商品SKU不存在
  - TQ3005：商品SKU编码已存在
  - TQ3006：商品库存不足
  - TQ3007：商品状态不允许操作
- **TQ4000-TQ4999**：进销存模块错误
  - TQ4001：入库单不存在
  - TQ4002：出库单不存在
  - TQ4003：盘点单不存在
  - TQ4004：库存预警不存在
  - TQ4005：盘点中，无法进行入库/出库操作
  - TQ4006：库存预警阈值最小值≥最大值
- **TQ5000-TQ5999**：会员模块错误
  - TQ5001：会员不存在
